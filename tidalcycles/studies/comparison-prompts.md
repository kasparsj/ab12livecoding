# Cross-Study Comparison Prompts

Use these prompts after students have tried at least one Kindohm study and one
ndr-brt study.

## Same Problem, Two Styles

| Prompt | Kindohm angle | ndr-brt angle |
| --- | --- | --- |
| Build a four-cycle transition. | Use `whenmod`, `trunc`, `stut`, and a return. | Use a section state and destabilize only one layer. |
| Make one sample feel like a whole track. | Scan and curate sample indices. | Build anchor, slow body, grain body, and accent body. |
| Create controlled instability. | Chain periodic transformations. | Restrict randomness to one dimension. |
| Build a melody from generated numbers. | Map `run` or choices into a scale. | Use a pseudo-Markov pattern or register layers. |
| Make a performance ending. | Drop layers with predictable cycle logic. | Rehearse section-state subtraction and global color change. |

## Reflection Questions

- What stayed fixed in each version?
- Which version was easier to recover from?
- Which version was more readable after five minutes?
- Which version sounded more performable?
- What would you remove before playing this live?

## Remix-Performance Architecture

| Prompt | Compare |
| --- | --- |
| Build a four-stem remix sketch from stock sounds. | How do named stems make hook, build, percussion, and bass roles easier to mute or return? |
| Route each stem to its own orbit with conservative gain. | Which layers need independent effects or recovery controls? |
| Move through full, breakdown, and return states. | Which state changes are readable enough to perform under pressure? |

## Hardware-Control Performance

| Prompt | Compare |
| --- | --- |
| Write the stock-sample fallback before any MIDI translation. | Does the musical idea survive without hardware? |
| Add `cF` or `cI` defaults to one transformation. | What changes when controller input is absent, slow, or noisy? |
| Translate one drum role to a generic MIDI channel. | Which details are musical roles, and which are device setup? |

## Drum-Structure Translation

| Prompt | Compare |
| --- | --- |
| Apply the same boolean gate to `bd`, `cp`, `hh`, and `arpy`. | What belongs to rhythm structure, and what belongs to sample choice? |
| Rebuild one groove family with only `struct` and stock sounds. | Which parts are essential to recognition? |
| Make a ghost-note version and a sparse version of the same gate. | Which version is easier to explain after five minutes? |

## Album/Project Organization

| Prompt | Compare |
| --- | --- |
| Create a boot-first project folder for one track. | What should live in shared setup versus the track file? |
| Fill out a sample manifest for stock, recorded, and extracted sounds. | Which samples are release-ready, and which need permission notes? |
| Write performance cues and archival hardware notes. | Which notes help a future performer, and which only document history? |

## Capstone Variants

- Five-day sketchbook: make five small files from one seed rhythm.
- Single-sample set: build a complete performance from one sample family.
- No-new-samples arrangement: use only `bd`, `sn`, `hh`, `cp`, and one melodic sample.
- One-pattern transformation performance: change form without replacing the base pattern.
- Stock remix architecture: arrange hook, build, percussion, and bass roles
  without upstream audio or commercial stems.
- Hardware-optional set: perform stock-sample parts first, then add controller
  or MIDI translations only where they improve playability.
- Drum-gate atlas: compare three groove families using the same sample palette.
- Final-project packet: combine a track folder, manifest, cue sheet, and
  production notes without requiring Jarmlib or external hardware.
